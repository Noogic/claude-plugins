# Ticket Template

> Each ticket is a detailed implementation spec for one roadmap milestone.
> Generated by `/forge-ticket [milestone]` or written manually.
> Consumed by `/forge-dev` (implementation) and `/test-dev` (testing).

## Structure

```markdown
# M[N]: [Milestone Title]

**Status**: draft | approved | in-progress | done
**Milestone**: [number and name from ROADMAP.md]
**Depends on**: [what must exist — specific models, migrations, pages from prior milestones]
**Produces**: [what exists after this is done that didn't exist before]
**Assumptions**: [gaps filled with defaults — e.g., "Assumed modal for create form (spec didn't specify)"]

## Decisions

> Only include this section if there are significant gaps that need user input.
> Each question includes options and a recommendation.
> The orchestrator presents these to the user before approving the ticket.
> Once resolved, fold answers into the relevant sections and remove this header.

### [Question title]
**Context**: [why this matters]
**Options**:
- A) [option] — [tradeoff]
- B) [option] — [tradeoff]
**Recommendation**: [which and why]

## Goal

> One paragraph. What is true after this milestone that wasn't true before?
> Written from the USER's perspective — what they can see and do.
> Example: "After login, the user sees a workspace selector. They can create
> a new workspace, switch between workspaces, and see the workspace name
> in the sidebar."

## Behavior

> Given/When/Then scenarios. These are the source of truth.
> If the code doesn't match these, the code is wrong.
> Each scenario maps to a test case.

### [Feature or interaction name]

**Given** [precondition]
**When** [action]
**Then** [expected result]

**Given** [error precondition]
**When** [bad action]
**Then** [expected error]
```
